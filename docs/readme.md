# Как бы я сформулировал вопросы эксперту (чтобы он помог заполнить таблицу правильно)

Ниже — вопросы “как техлид к эксперту”, без требования читать весь код, а именно на то, что важно для интеграции и чтобы не плодить дубли.

### A) По каждому модулю (универсальный чеклист)
1) **Это анализатор или исполнитель изменений?**  
   (Только читает/считает или реально меняет файлы?)
2) **Где точка входа?**  
   (CLI команда / Python API / класс / функция)
3) **Какие артефакты гарантированно создаются?**  
   (JSON/MD/patches/db) и где лежат
4) **Есть ли режим dry-run?**  
   Если есть — что именно он гарантирует (не пишет ничего / пишет только отчёты / пишет патчи)?
5) **Какие форматы стабильны?**  
   JSON в stdout строго или смешивается с логами? (очень важно для GUI)
6) **Как устроена безопасность?**  
   Есть ли backup/rollback? На сколько файлов? Транзакция или частично?
7) **Какие exclude/include правила используются?**  
   Что автоматически исключается (venv, unified, generated, .intellirefactor)?
8) **Где дублирование с другими частями?**  
   (backup/rollback/validation/fingerprints/patch engine)
9) **Какие самые частые причины падений/нестабильности?**  
   (память, recursion, json, import errors, циклы)
10) **Какая “зона ответственности” этого модуля в будущем?**  
   (оставляем как есть / делаем адаптер / переносим в общий слой)

### B) Вопросы специально для интеграции decomposition
1) Как лучше связать decomposition с audit/index?  
   - можно ли взять список файлов/символов из index.db, чтобы decomposition не сканировал всё сам?
2) Какие операции в системе считаются “разрешёнными в safe mode”?  
   - wrappers/unified/import rewrite/rename/move?
3) Нужно ли unified-папку считать “постоянной” частью проекта или временной?  
   - если постоянной: кто поддерживает её чистоту (импорты/коллизии)?
4) Какой механизм “единого rollback” предпочтительнее: safety_manager или decomposition apply engine?  
   (выбрать один как ядро, второй адаптировать)
